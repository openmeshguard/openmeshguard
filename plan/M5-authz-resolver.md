## Summary

### Decisions

- Authorization resolution uses Istio's additive policy model: applicable root-namespace and namespace policies form one union, ordered in the evidence chain as CUSTOM, DENY, then ALLOW. Selector mismatches remain explicit exclusion steps rather than disappearing as if no policy resource existed.
- An ALLOW policy with no `rules` is the default-deny idiom; `rules: [{}]` is an explicit broad allow. A default-deny ALLOW plus explicit ALLOW rules resolves to `default-deny-explicit-allow`; a namespace ALLOW without that idiom remains `allow-only`; any applicable DENY resolves to `deny-present` without discarding additive ALLOW evidence.
- CUSTOM is recorded ahead of native policies. CUSTOM plus a native posture uses the native effective enum and retains the CUSTOM chain step; CUSTOM-only remains explicit `unknown` because the external provider's result cannot be represented by the frozen enum.
- Ambient handling models attachment and enforceability only. Any targetRef policy needs a selected, observed, Programmed waypoint, whether its rules use L4 or L7 attributes; missing or unready attachment resolves to `waypoint-policy-unenforced`, unavailable Gateway evidence resolves to `unknown`, and selector-based L7 policy at ztunnel is recorded as Istio's fail-safe DENY. Ambient membership detection remains the M6 stub and is not inferred in M5.
- Before release, the human approved correcting the frozen authorization names from L7-specific `l7-policy-unenforced`/`l7Unenforced` to attachment-specific `waypoint-policy-unenforced`/`waypointUnenforced`. No compatibility alias is retained because there is no released contract, and the resolver/control versions identify the semantic change.
- The typed collector now gathers Services, EndpointSlices, DestinationRules, Sidecars, AuthorizationPolicies, and Gateway API Gateways with bounded list calls and per-scope availability. The normalizer uses EndpointSlice target references for selectorless-Service attachment, keeps observed-empty ports distinct from unavailable evidence, and propagates EndpointSlice/Pod gaps to workload and inventory availability.
- DestinationRules are resolved per observed client proxy: workload selectors and Sidecar egress scope use client labels, each client follows client-namespace → service-namespace → mesh-root lookup precedence, and only effective winners are unioned across clients. Port-level settings are translated from destination Service ports to workload target ports before reaching the pure resolver.
- AuthorizationPolicy projection reads both the pinned singular `targetRef` and plural `targetRefs`, emits one view per matching attachment, and retains target kind, name, and the exact Service/Gateway waypoint. TargetRef policy is never attributed to a sidecar; unknown or mixed data-plane membership yields explicit authorization `unknown`; `selector: {}` matches all workloads in scope; and ruleless DENY/CUSTOM never activate posture state.
- MG-AUTHZ-004 now consumes the approved `authorization.identityScoped` signal instead of duplicating MG-AUTHZ-003. Omitted `from`, empty sources, and prefix/suffix/presence wildcards are broad; every effective ALLOW branch must contain an explicit principal, namespace, service account, or request principal to be identity-scoped. An empty default-deny ALLOW is vacuously scoped only when no explicit ALLOW branch grants access.
- A post-review framework pass maps every authorization control to NIST CSF 2.0 `PR.AA-05`, NIST SP 800-53 Rev. 5 `AC-3`, and OWASP Kubernetes 2025 `K05`. Only MG-AUTHZ-002..004 also map to `AC-6` because they directly test least-privilege structure. OWASP `K02` is intentionally excluded because its published scope is Kubernetes API authorizers and RBAC, while `K05` covers workload traffic segmentation; weaker AC-4, SC-7, and CIS associations were not forced: [NIST CSF 2.0](https://www.nist.gov/publications/nist-cybersecurity-framework-csf-20), [NIST SP 800-53 Rev. 5](https://csrc.nist.gov/pubs/sp/800/53/r5/upd1/final), [OWASP Kubernetes K02](https://owasp.org/www-project-kubernetes-top-ten/2025/en/src/K02-Overly-Permissive-Authorization-Configurations.html), [OWASP Kubernetes K05](https://owasp.org/www-project-kubernetes-top-ten/2025/en/src/K05-Missing-Network-Segmentation-Controls).
- Review corrections preserve a DestinationRule port-level override even when that entry omits TLS, apply the root-namespace default Sidecar only when no namespace-local Sidecar applies, and split controller output to pod-level workloads when observed pods have different service/policy inputs.
- The resolver version is the stable ordered composite `mtls/v3,authz/v6`; only the subsystem tag whose semantics change is bumped. MG-AUTHZ-001..007 and MG-MTLS-007 ship as YAML/CEL data, and produced ports make MG-MTLS-002 evaluable.
- `port-level-override` and `dr-contradiction` now golden resolved evidence, their transitional notes are retired, and the new `sidecar-authz` cases cover root/local union, broad allow, DENY precedence, allow-only, and selector mismatch.
- The branch was rebased cleanly onto current `origin/main` at `abd6a49`; `a931d73` remains the first M5 commit and contains only the required Deliverables expansion.

### Flags raised

- CUSTOM → DENY → ALLOW ordering and root/local interaction were held for human review before tables were encoded. The approved result follows Istio's documented layered order and additive merge model, not PeerAuthentication's oldest-policy tie-break: [Istio authorization implicit enablement](https://istio.io/latest/docs/concepts/security/#implicit-enablement).
- Empty `{}` ALLOW versus `rules: [{}]` was held before encoding. Istio documents missing rules as never matching and an empty rule as always matching; the approved tables and `broadAllow` output preserve that distinction: [AuthorizationPolicy reference](https://istio.io/latest/docs/reference/config/security/authorization-policy/) and [allow-nothing/allow-all examples](https://istio.io/latest/docs/concepts/security/#allow-nothing-deny-all-and-allow-all-policy).
- Root-namespace policy interaction was held before encoding. Istio documents root policies as mesh-wide and selectors there as matching across namespaces; the approved resolver unions matching root and local policies and records selector exclusions: [AuthorizationPolicy scope](https://istio.io/latest/docs/reference/config/security/authorization-policy/).
- Ambient L7 boundaries were held before encoding. Istio documents targetRefs attachment to waypoints, ztunnel's inability to enforce L7, and selector-based L7 fail-safe DENY; the approved resolver distinguishes enforced, unenforced, and unavailable evidence: [Istio ambient L7 features](https://istio.io/latest/docs/ambient/usage/l7-features/).
- CUSTOM-only was held because Istio delegates the decision to the named external provider. The human approved `unknown` when no native effective class is available: [AuthorizationPolicy CUSTOM action](https://istio.io/latest/docs/reference/config/security/authorization-policy/).
- `broadAllow` was initially withheld, explained, and then explicitly approved as a structural signal for an applicable ALLOW containing an empty rule or wildcard principal/namespace. It is not a request-level simulation and remains omitted when authorization evidence is unavailable.
- The human explicitly approved the original frozen-contract edits: additive `AuthorizationPolicyView.HasSelector` and `HasRules`, `WaypointView.Known`, optional canonical `authorization.broadAllow`, and removal of unused `MeshDefaults.MeshMTLSMode`. During review remediation the human separately approved `AuthorizationPolicyView.IdentityScoped`, `TargetRefKind`, `TargetRefName`, `TargetWaypoint`, and optional canonical `authorization.identityScoped`. This follow-up review separately approved replacing the unreleased L7-specific unenforced names with `waypoint-policy-unenforced`/`waypointUnenforced` and adding authorization `not-in-mesh` for `ModeNotApplicable`.
- DestinationRule port overrides do not inherit omitted destination-level fields; review found and fixed the producer so an omitted port TLS block is retained as explicit unset/default evidence: [DestinationRule port-level settings](https://istio.io/latest/docs/reference/config/networking/destination-rule/).
- Root Sidecar defaults were added after review found that scoped scans collected only the workload namespace. Istio documents the root Sidecar as the fallback only when a namespace has no applicable local configuration: [Sidecar reference](https://istio.io/latest/docs/reference/config/networking/sidecar/).
- No RBAC profile was edited. Collection remains typed, bounded, get/list-only, and Secret-free; the action audit covers every new resource.
- Review finding 1 was confirmed and fixed by selecting DestinationRules and Sidecars from client proxy context. Istio defines `workloadSelector` as selecting the workload instances that receive the DestinationRule, including sidecars with egress TLS settings: [DestinationRule workloadSelector](https://istio.io/latest/docs/reference/config/networking/destination-rule/#DestinationRule).
- Review finding 2 was confirmed and fixed by merging the pinned typed API's singular `targetRef` with `targetRefs`, deduplicating them, and retaining attachment identity. Istio documents that target references determine which workloads/policies attach and that Service targets are waypoint-only: [AuthorizationPolicy targetRefs](https://istio.io/latest/docs/reference/config/security/authorization-policy/#AuthorizationPolicy).
- Review finding 3 was confirmed and fixed by translating each DestinationRule port selector through its Service `targetPort`. Istio defines a DestinationRule port selector as the destination Service port, while port-level policy overrides destination-level settings: [DestinationRule PortTrafficPolicy](https://istio.io/latest/docs/reference/config/networking/destination-rule/#PortTrafficPolicy).
- Review finding 4 was confirmed and fixed with typed EndpointSlice collection, availability, inventory, permissions, action audit, and selectorless-Service mapping. Kubernetes documents the `kubernetes.io/service-name` association and EndpointSlices as the source of truth for Service backends: [EndpointSlices](https://kubernetes.io/docs/concepts/services-networking/endpoint-slices/).
- Review finding 5 was confirmed and fixed with the documented client namespace → service namespace → mesh root lookup order instead of unioning every visible DestinationRule: [Istio traffic-management best practices](https://istio.io/latest/docs/ops/best-practices/traffic-management/#control-configuration-sharing-across-namespaces).
- Review finding 6 was confirmed and fixed with a distinct identity-scope signal. Istio states that omitted `from` permits any source and that strings support exact, prefix, suffix, and presence matching: [AuthorizationPolicy Rule](https://istio.io/latest/docs/reference/config/security/authorization-policy/#Rule).
- Review finding 7 was confirmed and fixed by binding each Service/Gateway targetRef policy to its exact waypoint and excluding targetRef policies from sidecar enforcement. Istio documents that waypoint authorization attaches through targetRefs to a waypoint or Service using it: [ambient L7 attachment](https://istio.io/latest/docs/ambient/usage/l7-features/#considerations-for-authorization-policies).
- Review finding 8 was confirmed and fixed by returning `unknown` before L4/L7 classification whenever data-plane membership is unknown or mixed, preserving the repository's hard unknown-propagation guarantee.
- Review finding 9 was confirmed and fixed by preserving `HasSelector=true` while treating an empty label selector as matching every object: [Kubernetes LabelSelector API](https://kubernetes.io/docs/reference/kubernetes-api/definitions/label-selector-v1-meta/).
- Review finding 10 was confirmed and fixed by letting ruleless ALLOW activate default deny while ruleless DENY/CUSTOM never match. Istio states that omitted rules never match and only explains the default-deny consequence for ALLOW: [AuthorizationPolicy rules](https://istio.io/latest/docs/reference/config/security/authorization-policy/#AuthorizationPolicy).
- Automated follow-up finding 1 was confirmed by a regression table that returned `allow-only` for an L4 Service-targeted policy with no waypoint before the fix. Waypoint resolution now follows `targetRefs` attachment independently of L4/L7 rule attributes; ready, missing, unready, and unavailable evidence have exact chain assertions. The human approved the breaking pre-release contract rename to `waypoint-policy-unenforced` and `waypointUnenforced`: [Istio ambient authorization attachment](https://istio.io/latest/docs/ambient/usage/l7-features/#considerations-for-authorization-policies) and [choosing L4 enforcement points](https://istio.io/latest/docs/ambient/usage/l4-policy/#choosing-enforcement-points-when-waypoints-are-introduced).
- Automated follow-up finding 2 was confirmed by denying Deployment discovery while retaining a client-selected plaintext DestinationRule: before the fix the producer emitted known-empty DestinationRule evidence and a known false contradiction. Client proxy evidence now becomes unavailable if any Pod, Deployment, ReplicaSet, StatefulSet, or DaemonSet list is incomplete, so MG-MTLS-007 resolves unknown instead of passing on partial clients.
- Automated follow-up finding 3 was confirmed from the `origin/main` resolver diff and the inherited-port table: M5 now emits every observed workload port in `byPort`, which changes mTLS output interpretation. The composite version regression failed at `mtls/v2,authz/v3` before the fix; the mTLS tag is now `mtls/v3`.
- Automated follow-up finding 4 was confirmed with one policy targeting two Services that select the same workload: before the fix both AuthorizationPolicy chain steps were identical `spec.action` entries. TargetRef-backed policy steps now identify the exact `spec.targetRefs["Kind/name"]` attachment, preserving ordered Service and waypoint evidence without duplicating `policiesInScope`.
- Automated follow-up finding 5 was confirmed by an unsupported `DELEGATE` action: before the fix the result was `unknown` but its chain contained only `AuthorizationDefault` and `policiesInScope` was empty. Unsupported in-scope actions now append the causative policy and exact `spec.action` evidence before returning unknown.
- Automated follow-up finding 7 was confirmed as a coverage gap: the prior tables separately covered CUSTOM+ALLOW and DENY+ALLOW but no single case asserted the complete CUSTOM → DENY → ALLOW chain. The added table fixes all four step orders, kinds, fields, and effects and resolves to the already approved `deny-present` classification.
- Automated follow-up finding 8 was confirmed as a coverage gap: no resolver table directly combined `RootNamespace`, a matching selector, and a workload in another namespace. The added cross-namespace root-selector table locks the approved mesh-scope behavior and its exact ALLOW chain: [AuthorizationPolicy scope](https://istio.io/latest/docs/reference/config/security/authorization-policy/).
- Automated follow-up finding 9 was refuted by the approval and baseline evidence. The M5 Deliverables explicitly require nil versus observed-empty ports; the frozen-change record explicitly approves `WaypointView.Known`; and `origin/main` returned authorization `unknown` for every input because M5 had not implemented authorization resolution, so no prior caller could rely on nil `AuthzPolicies` meaning collected-empty posture. The M5 no-policy and unavailable-evidence tables intentionally establish the new resolver's distinction.
- Automated follow-up finding 10 was confirmed: `ModeNotApplicable` with an ALLOW policy returned `allow-only` before the approved fix. Authorization now resolves `not-in-mesh` before policy availability, emits only the DataPlane membership chain, and omits policy-derived signals because out-of-mesh workloads have no Istio enforcement point: [Istio ambient data-plane workload categories](https://istio.io/latest/docs/ambient/architecture/data-plane/).
